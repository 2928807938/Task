# 项目架构、功能与逻辑梳理（无代码版）

## 1. 文档目标与范围

本文基于当前仓库代码结构与实现现状，对项目的以下内容做统一梳理：

- 架构设计：模块分层、技术选型、运行时组件关系
- 功能设计：当前已实现的业务域与接口能力
- 逻辑设计：关键业务流程、权限机制、状态流转、事件处理

说明：本文件仅做系统级说明，不包含代码实现细节。

## 2. 项目总体定位

这是一个面向团队协作场景的任务管理与需求分析系统，核心能力包括：

- 用户认证与账户管理
- 团队与项目管理
- 任务拆分、分配、依赖、状态流转
- 项目统计与仪表盘
- 基于 LLM 的需求分析与任务规划
- 邀请、通知、协议条款等配套能力

## 3. 分层与模块架构

## 3.1 多模块结构

项目采用 Gradle 多模块组织，核心模块如下：

- `shared`：通用能力（统一响应、异常、注解、上下文）
- `domain`：领域层（领域模型、领域服务、规则、事件、状态机、权限切面）
- `application`：应用层（用例编排、参数处理、VO 组装、流程控制）
- `infrastructure`：基础设施层（仓储实现、R2DBC、Redis、JWT、邮件、LLM 适配）
- `web`：接口层（Controller、WebFlux 安全过滤器、全局异常处理）
- `bootstrap`：启动层（Spring Boot 应用入口与装配）

## 3.2 典型调用链

一次典型业务请求的链路为：

1. `web` 接收 HTTP 请求，经过 JWT 与上下文过滤器
2. `application` 执行业务用例编排与参数转换
3. `domain` 执行核心业务规则、权限判定、状态流转与事件注册
4. `infrastructure` 完成数据库/缓存/外部服务读写
5. 结果回到 `web`，统一响应格式返回

## 3.3 架构风格与关键技术

- 语言与运行时：Kotlin + Java 21
- 应用框架：Spring Boot 3 + WebFlux（响应式）
- 数据访问：R2DBC + PostgreSQL（非阻塞）
- 缓存与会话辅助：Redis（验证码、Token 黑名单等）
- 鉴权：JWT + Spring Security（WebFlux）
- 领域风格：DDD 分层 + 注解式权限 + 事件驱动
- 事务事件：响应式事务性 Outbox（`ReactiveTransactionalOutbox`）
- 智能分析：OpenAI 兼容接口 + 内置 Prompt 体系

## 4. 功能域梳理

## 4.1 用户与认证

主要能力：

- 注册、登录、登出
- 当前用户信息获取
- 邮箱验证码发送与校验链路
- 修改密码
- 按邮箱/用户名检索用户（用于成员选择等场景）

支撑逻辑：

- JWT 签发与解析
- Redis 维护 Token 黑名单（登出后失效控制）

## 4.2 团队域

主要能力：

- 创建团队
- 当前用户团队列表检索
- 团队成员查询
- 团队首页数据（概览、活跃热力图、部门结构、近期沟通）

## 4.3 项目域

主要能力：

- 项目创建、更新、删除
- 项目成员增删
- 项目角色查看与自定义角色创建
- 项目状态列表、优先级体系查询
- 项目归档/取消归档
- 项目详情与分页列表
- 任务分布与状态趋势统计

项目创建时支持多种体系：

- 优先级体系：标准 / 高级 / 自定义
- 状态体系：标准 / 扩展 / 自定义（含状态转换规则）

## 4.4 任务域

主要能力：

- 按项目分页获取任务
- 任务详情、主任务及子任务整体视图
- 创建任务（主任务 + 子任务）
- 编辑任务（命令 + 策略模式）
- 更新任务状态
- 可用状态列表查询

扩展能力：

- 任务依赖关系维护（含循环依赖检查）
- 任务历史记录
- 任务进度与统计（完成率、趋势、分布）

## 4.5 需求分析与 LLM 域

主要能力：

- 需求分析流式输出（SSE）
- 任务智能分配流式输出（SSE）
- 需求对话列表的新增、查询、删除

分析链路包含：

- 需求分类
- 优先级分析
- 工作量分析
- 任务拆分
- 完整度检查
- 智能建议
- 综合总结

## 4.6 邀请、通知与协议

主要能力：

- 邀请链接创建、验证、禁用、按用户/项目查询
- 通知实体与偏好配置（模型与仓储已具备）
- 协议条款（管理端维护、客户端按 key 查询）

## 4.7 运维与可观测

主要能力：

- 健康检查端点
- 统一异常处理与错误码返回
- TraceId 贯穿请求日志

## 5. 核心业务逻辑设计

## 5.1 权限逻辑：RBAC + 注解式 AOP

系统采用“项目维度 RBAC + 注解式权限校验”：

- 通过 `@RequireProjectPermission` / `@RequireTaskPermission` 在应用层声明权限
- 由领域层 `PermissionAspect` 统一拦截并校验
- 任务权限额外支持“任务创建者/负责人”的特殊访问路径
- 复杂场景由 `AccessControlService` 执行细粒度判断

结果：业务代码中的手动权限判断明显减少，权限规则集中且可维护。

## 5.2 项目创建主流程

高层流程：

1. 校验输入并构建项目配置
2. 创建项目实体
3. 初始化权限基础数据
4. 将项目所有者加入项目成员
5. 初始化项目优先级体系
6. 初始化项目状态体系与状态转换规则
7. 建立状态映射关系

该流程覆盖了“项目可用即完整可管理”的初始化闭环。

## 5.3 任务创建主流程

高层流程：

1. 校验与规范化请求（默认截止时间、依赖合法性）
2. 读取相关成员已有任务，执行排程（避免冲突）
3. 创建主任务
4. 创建子任务并建立父子关系
5. 建立子任务依赖关系
6. 返回主任务标识

特点：

- 任务创建不是简单插入，而是包含“排程 + 依赖 + 结构化落库”的组合流程。

## 5.4 状态流转逻辑

项目状态：

- 通过状态定义 + 转换规则驱动
- 支持标准、扩展、自定义三种状态体系
- 转换规则独立持久化，可按项目动态配置

任务状态：

- 更新前校验“目标状态是否允许转换”
- 同时校验操作者是否具备修改权限
- 写入状态变更历史并发布任务状态事件

## 5.5 需求对话与分析结果落库逻辑

需求对话不仅保存一条主记录，还会串联（或并行）维护关联分析实体：

- 分类
- 完整度
- 优先级
- 建议
- 摘要分析
- 任务拆分
- 工作量

删除需求对话时执行级联删除，保证数据一致性。

## 5.6 事件驱动与 Outbox 逻辑

系统在关键领域操作上使用事件机制（尤其是项目相关操作）：

- 项目删除会触发相关领域事件
- 监听器分别处理任务清理、附件清理、成员通知等后续动作
- 通过响应式事务 Outbox，降低“主事务提交与事件发布不一致”的风险

## 5.7 安全与上下文传播逻辑

- JWT 过滤器负责认证与用户身份注入
- 请求上下文过滤器负责 TraceId/用户上下文透传
- 下游业务通过响应式上下文读取当前用户信息

## 6. 数据模型视角（按领域）

主要表可分为以下几组：

- 用户与认证：`t_user`、`t_user_profile`、`t_user_oauth`、`t_user_role`
- 权限与角色：`t_role`、`t_permission`、`t_role_permission`
- 团队域：`t_team`、`t_team_member`、`t_team_role`、`t_team_role_permission`
- 项目域：`t_project`、`t_project_member`、`t_project_role`、`t_project_role_permission`、`t_project_status*`、`t_project_archive_record`
- 任务域：`t_task`、`t_task_dependency`、`t_task_history`、`t_task_comment`、`t_task_tag`、`t_time_entry`、`t_time_estimation`
- 需求分析域：`t_requirement_*`
- 协同与审计：`t_activity_log`、`t_change_log`、`t_collaboration_record`、`t_document_version`
- 事件与通知：`t_domain_event`、`t_event_processing_log`、`t_notification*`
- 邀请与协议：`t_invite_*`、`t_agreement_term`

## 7. 当前架构特点总结

- 结构清晰：模块边界明确，职责分离较好
- 业务完整：围绕“团队-项目-任务-分析”形成主干闭环
- 机制较全：权限、状态、事件、审计、通知均有体系化设计
- 响应式一致性：Web、数据访问、外部调用都采用响应式风格
- 扩展友好：自定义状态流转、优先级体系、分析链路都支持继续扩展

## 8. 可持续演进关注点（架构层）

- 邀请模块目前仍有“待调试”痕迹，建议补齐端到端验证与统一响应风格
- 项目删除同时存在主流程清理与监听器清理，需持续关注重复执行的幂等策略
- 生产敏感配置（密钥、口令）建议统一迁移到安全配置中心或环境变量体系
- 可进一步补充“领域事件处理失败重试与补偿”可观测机制
